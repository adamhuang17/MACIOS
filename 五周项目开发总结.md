三、过程复盘与沉淀（必填）
1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？
本阶段主要完成了四类工作：方向收敛、架构设计、工程计划审查和 M0 底座实现。
首先，对比赛题目和评分场景进行了重新拆解。最开始项目容易被“多 Agent”“DAG”“RAG”“本地模型”“离线协同”“多端原生”等技术点牵引，但经过复盘后明确：比赛真正需要展示的不是一个通用 Agent 框架，而是一条从飞书 IM 到文档、PPT、归档、回传和状态同步的完整办公闭环。因此，项目从“通用多 Agent 编排系统”收敛为“飞书原生工作空间 Agent”。
其次，对现有 Agent-Hub 项目做了系统盘点。现有项目的优势在于已有较成熟的 Agent 引擎层，包括 DecisionRouter、AgentPipeline、ToolRegistry、SSE、PromptGuard、RAG、Memory、ReflectionAgent 等能力；短板在于缺少比赛所需的办公业务层，包括 Workspace、Task、Approval、Artifact、飞书连接器和可展示的 Web/PWA 状态中心。基于这个判断，最终决定不重写旧 Pipeline，而是在其上新增 agent_hub.pilot 作为比赛业务层。
然后，完成了目标架构和实施路线的收敛。项目被拆成“稳定内核 + Pilot 业务层 + 飞书/文档/PPT 技能层 + Web/PWA 展示层”的结构。旧 Pipeline 被重新定位为通用 Agent provider、/chat 兼容入口、失败兜底通道和回归基线；Pilot 则专门承载 Workspace、Task、Plan、Approval、Artifact、ExecutionEvent 等比赛业务对象。
接着，对两周路线图做了工程风险审查。原计划中一些能力虽然有亮点，但不适合放进主演示路径，比如飞书原生 API、PWA 离线合并、复杂本地/云模型动态路由、多端原生应用等。最终把方案拆成三类：主路径必须稳定，加分项可以展示技术深度，兜底路径用于保证演示不断。
最后，完成了 M0 底座的实际落地。M0 不只是建几个模型文件，而是把后续所有模块都会依赖的底层语义先钉牢：领域实体、状态枚举、纯函数状态机、ExecutionEvent、Plan DAG 校验、InMemory/SQLite 双 EventStore、进程内 EventBus 和共享契约测试。这样后续做编排、审批、SSE、飞书接入和产物中心时，可以直接复用稳定底座。
方法和工具上，主要用了以下几类：
- 代码盘点：先确认现有系统哪些能力成熟，哪些业务层缺失。 
- 竞品对标：用 OpenClaw、Microsoft 365 Copilot、Manus 校正项目边界。 
- 场景倒推：从评委演示场景反推架构，而不是从技术组件正推产品。 
- 风险拆解：把功能分成主路径、加分项和兜底路径。 
- AI 协同：用 AI 辅助拆题、对比、归纳、生成架构草案、状态迁移表和测试样例。 
- 工程验证：使用 pytest、Pydantic、asyncio、SQLite 等完成 M0 的测试和回归验证。 
AI 的主要作用是加速信息整理和方案展开；人工侧主要负责产品判断、scope 收敛、架构边界、状态语义、事务边界和最终技术取舍。

2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？
本阶段主要有四个卡点。
第一个卡点是：项目能力很强，但太通用，难以直接对应比赛评分点。
之前实现的项目已经具备多 Agent、DAG、ReAct、RAG、记忆、安全、流式响应等底层能力，但如果继续沿着“通用 Agent 框架”方向推进，评委很难快速感知完整办公闭环。破局方式是先定唯一主链路，只保留最能证明比赛价值的闭环：飞书 IM 发起 → Workspace/Task → Plan → Approval → Context → Doc → Slides/PPTX → Drive → 群聊回传 → Web/PWA 状态同步。这样项目从“什么都能做的框架”变成了“能稳定演示的飞书办公 Agent 产品”。
第二个卡点是：scope 一度过大。
原先同时想做多可见 Agent、复杂图谱、离线协同、GUI、原生移动端、文档解析、PPT 生成、飞书 Slides、复杂模型路由等能力，两周内很容易失控。破局方式是做取舍：对外只保留一个统一助手；多端先用 Web Dashboard + PWA；离线 CRDT 后置；PPT 主路径走 Brief Markdown → SlideSpec JSON → PPTX/Drive；飞书 Slides、卡片审批和复杂模型路由作为加分项或 fallback，不阻塞主链路。
第三个卡点是：旧 Pipeline 到底该不该动。
一开始想着既然重构，那之前旧的编排就需要删除，但后续经过与AI的交流，觉得以旧 Pipeline 为稳定引擎层，Pilot 作为比赛业务产品层，旧 Pipeline 可以作为可复用的 provider、兼容入口和演示兜底。
第四个卡点是：M0 实现时，状态语义容易和存储层混在一起。
最开始很容易让存储层顺手做状态判断，或者让业务模型和 IO 混在一起。后来直接把规则定死：状态变更只能走纯函数 transition，存储层只负责 append 和 read，不做业务判断。这个边界明确后，领域模型、状态机、EventStore 和测试都变得更清晰。另一个具体问题是 EventBus 的溢出行为，原先队列一满就立即抛异常，导致已经进入缓冲区但还没消费的消息也读不到。后来改成“先允许排空缓冲消息，再在下一次读取时抛出溢出异常”，行为更符合真实系统预期，测试也更稳定。

3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？
这次我觉得最重要还是一套对于比赛需求的定位方法、通用项目的转变过程等，下次遇到同样的情况可以复用。
第一套是 “先产品对象，后 Agent 编排”。先定义 Workspace、Task、Plan、Approval、Artifact、ExecutionEvent 这些业务对象，系统才像一个产品；否则无论底层 Agent 多强，用户看到的仍然只是一个聊天框。
第二套是 “先锁主链路，再谈高级能力”。先把 IM → Doc → Slides/PPTX → Drive → 群聊回传 → Dashboard 跑通，再考虑复杂多 Agent、图谱、离线协同、插件市场和富媒体编辑。
第三套是 “主路径 / 加分项 / 兜底路径”三分法。主路径必须稳定、简单、可录屏、可恢复；加分项可以体现技术深度，但不能阻塞主流程；兜底路径必须能在断网、API 失败、LLM 超时的情况下继续演示。
第四套是 “高风险写操作默认 dry-run + approval”。凡是涉及文档写入、PPT 生成、飞书云盘上传、群聊回传的动作，都应该先进入计划和审批，再执行。这个模式以后做任何自动执行型 Agent 都能复用。
第五套是 “文档作为 PPT 的中间层”。不直接从聊天生成 PPT，而是先生成 Project Brief，再生成 SlideSpec，最后渲染到 PPTX 或飞书 Slides。这样稳定性、可修改性和可解释性都更强。
第六套是 “前端只订阅业务事件流”。只要有 ExecutionEvent，Web Dashboard、PWA、飞书卡片、后续桌面端或移动端都可以同步展示任务状态，不需要前端理解 router、pipeline、tool agent 等内部细节。
第七套是 “先状态机，后存储”。先把所有状态迁移写成明确规则，再让实体保持不可变，只通过 transition 产出新快照；存储层只负责事件追加和读取。这个方式后续接审批、重试、恢复点和审计都更稳。
第八套是 “共享契约测试”。InMemory 和 SQLite 不各写一套逻辑说明，而是跑同一组测试，确保 demo mode 和默认持久化实现行为一致。后续如果接 Redis EventBus、Postgres EventStore 或真正的服务层，也可以继续复用这套测试思路。

我自己本人使用Openclaw时，基本都接着云端模型进行使用（虽然openclaw也支持本地部署），但在一些弱功能场景下，不仅速度慢效率低，没有发挥出很好的效果。如果能够接入一个本地模型，作为弱功能的处理核心，放在低风险、高频、隐私友好、弱网可用的固定能力层里，用来做意图粗分、上下文压缩、字段抽取、安全预筛和 demo fallback，实现一种“本地优先、云端兜底、低成本可控”的双速执行策略，这是目前OpenClaw“不原生支持”的。

最终这套项目才可以用一句话概括：
我们不是给办公软件加一个 AI 按钮，而是让 Agent 成为任务主驾驶：飞书 IM 是入口，Web/PWA 是仪表盘，文档和 PPT 是可追踪、可审批、可恢复、可迭代的交付物。

1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？
本周期主要完成了四类工作：服务编排、技能系统、API/Dashboard、离线验证。
首先，把第一周期的领域对象接到了真实服务层里。M0 已经有 Workspace、Task、Plan、PlanStep、Approval、Artifact 和事件存储，但还缺一个业务入口来驱动它们。本周期新增 TaskOrchestrator，负责统一处理任务提交、Workspace 创建或复用、幂等去重、Plan 生成、审批申请和执行触发。这样外部入口不需要理解底层状态机，只需要提交 TaskRequest。
其次，完成了 Planning、Approval 和 Execution 的职责拆分。PlanningService 负责把规划蓝图物化为 Plan 和 PlanStep，并继续复用 M0 的 DAG 校验；ApprovalService 负责 Plan 和 Step 两级审批；ExecutionEngine 负责按依赖顺序执行 step，并在 write/share 类步骤前做 dry-run 和审批门控。这个拆分让“计划怎么来”“审批怎么走”“步骤怎么跑”变成了三件清楚的事情。
然后，做了 Pilot 专用 SkillRegistry 和 fake skills。旧项目里已经有 ToolRegistry，但它更偏通用工具调用；Pilot 需要的是带业务语义的技能，包括 side_effect、requires_approval、supports_dry_run、timeout、scope 等元信息。因此本周期单独建立了 SkillRegistry，并用 fake skills 跑通 context、brief、slidespec、pptx、upload、summary 这条演示链路。
接着，完成了 M2 的 API 和 Dashboard。API 层不直接碰状态机，而是通过 CommandService 和 QueryService 转发。QueryService 聚合 TaskSummary、TaskDetail、WorkspaceSummary，供 Dashboard 展示；CommandService 集中处理审批决议、拒绝后的任务终止、重试和恢复。Dashboard 虽然是最小单页实现，但已经能看到任务列表、步骤状态、审批按钮、工件列表和 SSE 实时事件流。
最后，补了离线验证思路。offline_verify 脚本把核心 Pipeline 和 Pilot fake 链路一起跑起来，用 fake router、fake LLM、fake skills 替代外部依赖。这个验证对比赛很重要，因为演示不能完全押注在真实飞书、真实 LLM、真实 Drive API 都稳定可用上。
方法和工具上，主要用了以下几类：
代码分层：把业务入口、规划、审批、执行、查询、命令拆开，不让 API 层承载业务判断。
状态机复用：所有状态变化继续通过 transition，不在服务层手写状态跳转。
事件驱动：服务层统一写 ExecutionEvent，再由 EventBus 推给 Dashboard。
兜底优先：先用 fake skills 和 offline_verify 保证链路可跑，再为后续真实飞书和文档/PPT 链路留接口。
AI 协同：用 AI 辅助展开服务类、API DTO、测试边界和 Dashboard 最小交互；人工负责最终边界判断、状态语义和演示优先级。
2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？
本周期主要有四个卡点。
第一个卡点是：M0 的领域底座很完整，但还不像一个能用的产品。只有模型、状态机和事件存储时，系统是“正确的”，但用户仍然不知道从哪里提交任务、在哪里审批、在哪里看结果。破局方式是引入 TaskOrchestrator 作为统一业务入口，再补 CommandService 和 QueryService，让写操作和读模型都有明确边界。
第二个卡点是：审批逻辑容易和执行逻辑缠在一起。Plan 审批、Step 审批、自动审批、手动审批、拒绝后失败、通过后继续执行，这些状态很容易写成 if/else 泥团。后来把规则收敛成两级审批：Plan 审批决定任务能否开始，Step 审批控制有副作用的写入和分享动作。ExecutionEngine 只在步骤执行前触发审批门控，真正的审批决议统一交给 ApprovalService 和 CommandService。
第三个卡点是：Dashboard 的实时事件不能只靠“当前状态”。如果用户刷新页面，之前的事件可能已经错过；如果只订阅实时消息，又会丢历史轨迹。破局方式是 SSE 采用 backfill + live 的方式：先从 EventStore 回放历史事件，再切到 EventBus 实时订阅，同时用 sequence 做去重。这样 Dashboard 既能看当前状态，也能看到任务是怎么走到当前状态的。
第四个卡点是：真实外部能力还没准备好，但演示链路不能等。飞书、Drive、文档、PPTX、LLM 都有不确定性，如果 M1/M2 就强依赖它们，项目会被外部 API 拖住。破局方式是先做 fake skills，把“控制流正确”与“真实技能实现”解耦。只要 fake skills 能稳定跑通，后续替换成真实文档生成、PPTX 渲染和飞书上传时，就不会动摇主架构。
3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？
第一套可复用的是 “Orchestrator 作为业务入口”。外部入口只提交 TaskRequest，不直接操作 Plan、Approval、Step 和 Artifact。这个模式以后接飞书 webhook、Web 表单、CLI 或 API 都能复用。
第二套是 “Command / Query 分离”。审批、重试、恢复这类写操作集中进 CommandService；任务详情、列表、Dashboard 投影集中进 QueryService。这样前端不会被迫理解底层事件存储和状态机细节。
第三套是 “SkillSpec 元数据驱动”。每个 skill 都声明 side_effect、requires_approval、supports_dry_run、timeout 和 scopes，执行引擎就能按策略处理，而不是让每个技能自己决定是否需要审批。这对后续接真实飞书写入、Drive 上传、PPTX 渲染尤其有用。
第四套是 “fake skills 先打通控制流”。真实能力可以晚一点接，但任务链路、审批链路、事件链路和 Dashboard 必须先跑起来。这个思路以后做任何外部 API 很多的 Agent 项目都能复用。
第五套是 “SSE backfill + live”。只做实时推送不够，只查最终状态也不够；事件型 Dashboard 最好同时具备历史回放、实时更新和 sequence 去重。这个模式后续也可以给飞书卡片、PWA、桌面端继续复用。
第六套是 “演示兜底路径工程化”。offline_verify 不是临时脚本，而是把本地可验证能力固化下来：没有真实 LLM、飞书和数据库时，仍然能证明 Agent-Hub 的控制流、状态流和产物流是通的。
四、本周随手记（选填）
这一周期最大的感受是：Agent 项目真正变成产品，不是从接入更多模型开始的，而是从“任务能不能被追踪”开始的。
第一周期更像是在把规则钉牢：有哪些实体、状态怎么变、事件怎么存。第二周期则是在问另一个问题：用户发起任务之后，系统到底怎么接住它？谁来生成计划？谁来等审批？谁来执行？失败后从哪里恢复？评委打开页面能看到什么？
Dashboard 在这一阶段的价值也更清楚了。它不是为了做一个漂亮前端，而是为了让 Agent 的内部过程可见：当前任务状态、待审批项、步骤执行情况、产物数量、事件序列、失败恢复入口。这些信息一旦能展示出来，Agent 就不再像一个黑盒聊天机器人，而像一个有流程、有审计、有恢复能力的办公系统。
这周期也进一步确认了一个判断：真实飞书和真实 PPT 能力很重要，但它们不应该成为最早的风险中心。先用 fake skills 把业务骨架跑通，再逐步替换真实技能，项目会稳很多。比赛演示尤其如此，最怕的是功能看起来很高级，但关键路径因为一个外部 API 或权限问题断掉。
最终第二周期可以用一句话概括：
第一周期让 Agent-Pilot 有了可信的领域底座；第二周期让它第一次像一个能被用户操作和观察的办公 Agent 产品。

三、过程复盘与沉淀（必填）
1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？
本周期主要完成了四类工作：Artifact 内容化、上下游输入传递、内部文档技能、Brief 修订链。
首先，把 Artifact 从“元数据快照”升级成了“内容产物”。之前 skill 可以返回 artifact_payload，但系统主要记录 type、title、uri、metadata 等字段，缺少统一的内容落盘和读取机制。本周期新增 ArtifactStore，统一处理 content 字段，将 Markdown、JSON、二进制内容写入文件或内存，并记录 storage_key、uri、mime_type、checksum 等信息。
其次，解决了 step 之间无法传递真实产物内容的问题。文档链天然是上游依赖下游：Brief 需要读取 ContextBundle，修订 Brief 需要读取旧 Brief。为此新增 StepInputResolver，用 step 的 output_artifact_ref 建索引，把上游 Artifact 内容读出来后注入到下游 skill 的 input_artifacts 中。这样 ExecutionEngine 不需要理解每个技能的业务语义，只负责把“上游产物”按统一契约传下去。
然后，完成了内部文档技能。internal.context.assemble 用来把原始需求、附件引用、Markdown 引用和上游消息整理成 ContextBundle；internal.brief.generate 用 TemplateBriefGenerator 生成 Project Brief Markdown；internal.brief.revise 则读取旧 Brief，按修订指令生成新版 Brief。技能本身不直接读写仓库，而是通过 ArtifactStore 和 BriefGenerator 的依赖注入工作，这样后续可以把模板生成器替换成 LLM 生成器。
接着，补上了文档版本链。修订 Brief 时，新 Artifact 会带上 parent_artifact_id，并自动生成 artifact_version=2；旧版本会被标记为 SUPERSEDED。这个能力很关键，因为办公场景里的文档不是一次性输出，而是不断迭代的交付物。版本链让后续“重生成文档”“按意见修改”“回滚或对比”都有了基础。
最后，补充了 M3 专项测试。测试不仅覆盖 happy path，也覆盖了内容读取、磁盘/内存两种存储、上游 Artifact 注入、缺失输入失败、Brief v2 修订和旧版 supersede。这样本阶段不是只靠 demo 跑通，而是把文档链的关键语义都锁进了测试。
方法和工具上，主要用了以下几类：
契约先行：先确定 artifact_payload、input_artifacts、parent_artifact_id 的数据契约，再写具体技能。
依赖注入：BriefGenerator 和 artifact_reader 都通过注册函数注入，避免 skill 直接绑定仓库实现。
确定性模板：先用 TemplateBriefGenerator 保证离线稳定，再为后续 LLM 文档生成留替换口。
内容寻址：用 sha256 做内容校验，降低重复写入和路径不一致带来的风险。
测试驱动：围绕 ArtifactStore、StepInputResolver、internal skills 和 revision chain 写专项测试。
AI 协同：用 AI 帮助展开模板章节、梳理测试边界、定位跨平台文件路径问题；人工负责最终契约边界和状态语义。
2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？
本周期主要有四个卡点。
第一个卡点是：Artifact 到底应该存什么。最开始容易只把 uri、metadata、title 存进去，但这样下游 step 没法真正读取内容，Brief、SlideSpec、PPTX 之间就无法形成链路。后来把规则明确为：Artifact 必须同时包含元数据和可读取内容；content 可以落盘，也可以在测试中走 memory://。这样 Artifact 才真正成为交付物，而不是日志记录。
第二个卡点是：下游步骤如何拿到上游结果。不能让每个技能自己去仓库里猜，也不能让 ExecutionEngine 硬编码“Brief 读取 ctx，PPT 读取 spec”。破局方式是引入 StepInputResolver，用 output_artifact_ref 和 inputs_from 建立通用映射。只要 plan 里声明了 inputs_from，下游 skill 就能通过 input_artifacts 拿到对应内容。
第三个卡点是：文档修订不能强制走完整 PPT 六步链路。修订 Brief 只需要重新收集上下文和生成文档，如果复用完整模板，会把 SlideSpec、PPTX、上传、回传都拉进来，既慢又不符合语义。后来新增 BriefRevisionContext 和 generate_revision，把修订单独做成 READ_CONTEXT → GENERATE_DOC 的 mini-plan，避免为了复用而牺牲清晰度。
第四个卡点是：Windows 下 file:// URI 解析有坑。直接 unquote parsed.path 会得到类似 /D:/... 的路径，导致文件不存在。后来改用 url2pathname 处理 file URI，再转成 Path，解决了 Windows 路径兼容问题。这个问题虽然小，但如果不修，artifact 落盘后的读取会在本地演示时直接断掉。
另一个具体问题是版本链和快照版本号。旧 Artifact 被 supersede、新 Artifact 保存、父子关系更新，如果 expected_version 处理不好，很容易触发并发冲突。后来把新 Artifact 初次保存和旧 Artifact 状态迁移分清楚，该用 expected_version=None 的地方明确使用，状态变化仍然走 transition，边界就稳定了。
3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？
第一套是 “Artifact = 元数据 + 内容 + 版本”。以后所有产物都可以按这个模式处理：Brief、SlideSpec、PPTX、Drive 文件、分享链接都只是 Artifact 的不同类型。
第二套是 “input_artifacts 契约”。Step 不需要知道上游怎么生成，只要声明 inputs_from；ExecutionEngine 负责把内容注入，Skill 只消费输入。这套模式后续接 PPTX 渲染、飞书文档、摘要生成都能复用。
第三套是 “Template 作为 LLM 的稳定兜底”。TemplateBriefGenerator 虽然不如 LLM 灵活，但它稳定、可测、离线可跑，非常适合 demo fallback。以后可以继续保留“模板优先兜底，LLM 增强生成”的双路径。
第四套是 “修订任务用 mini-plan”。不是所有任务都应该走完整链路。修订、摘要、补充上下文、重新上传都可以是小计划。这个思路能避免系统变成一个所有场景都强行六步执行的僵硬流程。
第五套是 “父子版本链 + supersede”。办公文档天然会反复修改，parent_artifact_id 和 artifact_version 这套设计后续可以直接扩展到 SlideSpec、PPTX、飞书文档版本和 Drive 文件版本。
第六套是 “跨平台文件 URI 处理要尽早测”。尤其是在 Windows 开发环境里，file://、Path、相对路径、绝对路径都容易踩坑。以后凡是涉及文件落盘和 URI，都应该单独加 Windows 兼容测试。

三、过程复盘与沉淀（必填）
1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？
本周期主要完成了五类工作：飞书入口、飞书技能、意图分流、profile 化规划、真实产物链雏形。
首先，完成了飞书连接器的基础落地。飞书事件不是直接塞进业务层，而是先经过 FeishuWebhookProcessor 做验签、解密、去重、过滤和规范化。这样后续不管是 HTTP webhook 还是长连接事件，理论上都可以复用同一个 FeishuInboundMessage 契约。这个设计避免了把飞书原始 JSON 结构散落在业务服务里。
其次，完成了 FeishuWebhookService 到 TaskOrchestrator 的接线。服务层会按 feishu_chat_id 查找已有 Workspace，如果没有就创建新的 Workspace，再把消息包装成 TaskRequest 提交给 orchestrator。飞书 ACK 属于连接器自己的事务，不进入 ExecutionEngine，也不需要审批，避免“收到任务”这种轻量反馈被工作流复杂化。
然后，补齐了飞书技能。feishu.im.fetch_recent 用于拉取上下文，feishu.docs.create_doc 用于创建飞书文档，feishu.drive.upload_file 用于上传产物，feishu.im.send_message 用于回传摘要。更重要的是，这些技能没有绕过原来的 SkillRegistry，而是继续声明 side_effect、requires_approval、supports_dry_run 和 scopes。这样飞书写入动作自然接入了 Pilot 的审批体系。
接着，处理了 M4 暴露出的一个产品问题：所有飞书消息都进入任务链路是不合理的。普通问答、进度查询、真正的文档 / PPT 任务应该分开走。于是引入 PilotIngressService，用确定性关键词规则先区分 ignore、ordinary_qa、progress_query、start_task。普通问答仍然可以走旧 AgentPipeline，进度查询走 PilotQueryService，真正任务才进入 orchestrator。
随后，完成了 M5 的 profile 化规划。原来 TemplatePlanGateway 更像固定六步 PPT 模板：context、brief、slidespec、pptx、upload、summary。这个模板适合 PPT 任务，但不适合“写一份 brief”“总结一下会议”“修改一下旧文档”等场景。本周期新增 PlanProfile，把任务分成 deck_full、doc_only、summary_only、revision 四种形态，并让 PlanningService 按 profile 校验必需步骤。
最后，补了真实产物链的雏形。real.slide.generate_spec 能从 Brief 生成 SlideSpec JSON，real.slide.render_pptx 能用 python-pptx 渲染 PPTX，不可用时降级成 stub 字节，real.drive.upload_share 能把 PPTX 上传到飞书云空间并返回链接。这样主路径虽然仍保留 fake fallback，但已经具备向真实 PPT / Drive 链路切换的工程基础。
方法和工具上，主要用了以下几类：
分层接入：飞书原始事件只停留在 connectors 层，Pilot 业务层只接收规范化 DTO。
双重幂等：连接器层用 event_id 去重，orchestrator 层用 idempotency_key 去重。
渐进替换：fake skills 继续保留，real_chain 通过配置开关逐步替换。
profile 化模板：先用确定性规则和模板兜底，再允许 OpenAIModelGateway 生成 JSON plan。
审批复用：飞书写入、上传、分享不单独设计安全机制，而是复用 Pilot step 审批门控。
AI 协同：用 AI 帮忙展开接口契约、梳理飞书事件处理边界、生成测试矩阵；人工侧负责确定哪些能力进主演示、哪些只做加分项或后续任务。
2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？
本周期主要有五个卡点。
第一个卡点是：飞书事件格式复杂，不能直接让业务层处理。飞书有 URL verification、v1/v2 token、加密 payload、不同 event_type、群聊 mention、消息 content JSON 字符串等细节。如果直接在 service 或 API 路由里拆，会很快变乱。破局方式是把 FeishuWebhookProcessor 做成单独层，专门处理飞书协议细节，输出统一的 FeishuInboundMessage。
第二个卡点是：重复事件会导致重复任务。飞书 webhook 或长连接都有重投可能，如果每次收到都创建 Task，Dashboard 和后续产物会重复。破局方式是做双保险：webhook processor 层用 event_id 做 TTL LRU 去重，orchestrator 层继续用 idempotency_key 写入 ExecutionEvent 的唯一约束。这样即使连接器层漏掉，业务层也能兜住。
第三个卡点是：真实飞书写操作必须受控。创建文档、上传文件、发送群消息都属于有副作用动作，不能让模型自由调用。破局方式是把飞书能力封装成 SkillSpec，并把 write/share 技能默认标记为 requires_approval=True。模型或 planner 只能生成计划，真正执行前仍然经过审批门控和 dry-run。
第四个卡点是：固定六步模板开始限制产品体验。用户只是想写一份 brief，系统却生成 PPT；用户只是问“进度怎么样”，系统也可能误建任务。破局方式分两层：Ingress 先判断这是不是任务；如果是任务，再由 PlanProfile 决定它是 deck_full、doc_only、summary_only 还是 revision。这样“是否做任务”和“做什么类型任务”被拆开了。
第五个卡点是：真实 LLM 规划不稳定。OpenAI 兼容接口虽然可以输出 JSON，但模型可能返回非法 JSON、遗漏字段、使用不存在的 skill，或者网络失败。破局方式是坚持 fallback：OpenAIModelGateway 失败时回退 TemplatePlanGateway，PlanningService 继续做 profile 必需 step 校验，写 / 分享类 step 仍然强制 requires_approval。真实模型可以增强体验，但不能破坏主链路稳定性。
还有一个演示层面的卡点是：真实 PPTX、Drive、飞书分享链路依赖环境和权限。为了不让演示被 python-pptx 或飞书权限卡住，real.slide.render_pptx 在依赖缺失时生成 stub，fake skills 也继续保留。这样即使真实链路不完整，系统仍然能展示计划、审批、Artifact 和事件流。
方法和工具上，主要用了以下几类：
分层接入：飞书原始事件只停留在 connectors 层，Pilot 业务层只接收规范化 DTO。
双重幂等：连接器层用 event_id 去重，orchestrator 层用 idempotency_key 去重。
渐进替换：fake skills 继续保留，real_chain 通过配置开关逐步替换。
profile 化模板：先用确定性规则和模板兜底，再允许 OpenAIModelGateway 生成 JSON plan。
审批复用：飞书写入、上传、分享不单独设计安全机制，而是复用 Pilot step 审批门控。
AI 协同：用 AI 帮忙展开接口契约、梳理飞书事件处理边界、生成测试矩阵；人工侧负责确定哪些能力进主演示、哪些只做加分项或后续任务。
3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？
第一套是“连接器层标准化 DTO”。飞书、钉钉、企业微信、QQ 以后都可以走类似结构：原始事件只在 connector 里解析，业务层只接收统一 InboundMessage。
第二套是“ACK 不进入工作流”。收到消息后的“已收到任务”是连接器事务，不是 Agent 任务步骤。这个边界能让用户及时获得反馈，也避免把轻量确认消息放进审批和执行链路里。
第三套是“双重幂等”。外部平台事件重投非常常见，连接器去重和业务事件去重应该同时存在。这个模式以后接任何 webhook 都能复用。
第四套是“IngressIntent 和 PlanProfile 分离”。IngressIntent 判断“这条消息走什么入口”，PlanProfile 判断“任务应该生成什么计划”。这两个概念分开后，系统比单纯关键词路由清楚很多。
第五套是“profile-aware planning”。doc_only、summary_only、revision 这种短链路非常重要，不是所有任务都应该走完整 PPT 流程。以后可以继续扩展成 research_only、slides_only、upload_only、qa_only 等形态。
第六套是“真实链路 + fake fallback 并存”。比赛项目既需要真实能力证明，也需要稳定 fallback 保证演示不断。配置开关控制 skill_mode 和 gateway，是比较稳的工程策略。
第七套是“OpenAI JSON mode 后仍要二次校验”。模型输出 JSON 不等于业务可信，必须经过 Pydantic 校验、profile 必需步骤校验、skill 白名单校验和审批策略校验。
四、本周随手记（选填）
这个周期最大的变化是：项目终于开始真正进入飞书办公场景了。前面几轮做的是底座、流程、产物，现在开始面对真实平台事件、权限、消息格式、去重、回复、上传、文档创建这些“产品现场”的问题。
我也更明显地感受到，飞书接入不能只是“收到消息就调用 Agent”。真实办公群里有普通聊天、有问答、有进度追问、有任务委托、有文件和附件，还有各种重复事件和权限限制。Agent 如果不做分流，就会显得很冒失；如果不做审批，就会有安全风险；如果不做幂等，就会产生重复任务。
这周期另一个重要收获是：计划模板必须有弹性。固定六步 PPT 链路适合主演示，但不是所有需求都该这么重。一个好的办公 Agent 应该能轻重分流：简单问答快速回，进度查询直接查，brief 任务走短链，PPT 任务走完整链，修订任务只生成新版文档。
飞书原生能力也再次证明了一个设计判断：外部 API 不能裸露给模型。飞书 client 是底层能力，真正给 Agent 用的一定要是 skill；skill 要声明副作用、审批要求、dry-run 支持和 scope。这样 Agent 才不是“想调什么就调什么”，而是在一个可控的业务轨道上执行。
本周期可以用一句话概括：
第三周期让 Agent-Pilot 的产物真正可交付；第四周期让 Agent-Pilot 进入飞书协作现场，并开始根据不同任务类型生成合适的办公工作流。

原来的 Dashboard 更偏向开发调试视角，主要展示任务 JSON、审批 JSON、事件 JSON。本周期完成了整体重构，升级为“Agent-Hub 任务驾驶舱”：

这让演示从“电脑上能看”提升到“飞书、电脑、手机三端都能看”。尤其是审批场景，可以明确展示：同一个任务，在飞书卡片、桌面 Dashboard、手机 Dashboard 上是同步的。
成果区完成可读化

PPT 大纲 SlideSpec：可预览结构化 JSON，并显示页数。

飞书审批卡片链路补齐为可演示闭环
本周期把飞书审批从“理论上能审批”推进到“卡片能发、按钮能点、点完能刷新终态”的完整链路。
核心补齐点包括：
新增 FeishuApprovalNotifier，在 approval.requested 后自动发送飞书 interactive card。
审批卡片发送后持久化 channel_message_id，为后续更新卡片状态做准备。
长连接支持 card.action.trigger / p2.card.action.trigger。
飞书卡片按钮点击后会调用 PilotCommandService.decide_approval。
审批完成后，服务端会把原审批卡更新为最终态卡片，批准显示绿色，拒绝显示红色。
卡片回调返回 toast 和 replacement card，避免飞书客户端点完后又回到原始待审批状态。
审批人白名单校验已加上，非审批人不能越权决议。
这部分是第五周期很关键的产品化成果：审批不再只是 Dashboard 内部动作，而是真正变成飞书里的自然协作体验。
Step 审批后的续跑能力增强
本周期还补上了 step 级审批后任务继续执行的问题。之前 step approval 更像一个阻塞点，审批通过后还需要借助恢复任务或额外操作继续推进。现在支持：
step approve 后可以在原 task / plan 上自动续跑。
run_after_approval=false 时任务保持 blocked，可在 Dashboard 手动点击“继续执行”。
新增 resume task 能力，兼容历史上“已审批但未续跑”的任务状态。
step reject 后会明确把 task 置为 failed，避免任务处于含糊状态。
这让演示中的审批链路更自然：审批通过之后，系统继续往下跑；拒绝之后，任务明确终止。
PPT / Drive 真实产物链继续加固
本周期针对飞书 Drive 链路做了几个实用增强：
real.drive.upload_share 不再只接受 PPTX，也可以消费 brief/doc/file，上游是文档任务时也能上传分享。
上传后支持自动 share_file 给管理员 open_id，并触发飞书“与我共享”通知。
当飞书 upload_all 不返回 download_url 时，会用 feishu_drive_url_template 基于 file_token 拼出可访问链接。
share_url、file_token、feishu_token 等元数据统一写入 artifact，方便 Dashboard 和飞书消息复用。
DOC_ONLY profile 也从 ctx → brief 扩展为 ctx → brief → upload → summary，让文档类任务也能最终回传可访问链接。
这部分解决的是外部平台 API 的不确定性：飞书 Drive 不一定直接给下载链接，所以系统需要自己做兜底拼接和权限分享。
演示 Runbook 固化
本周期新增并固化了比赛演示 Runbook，把黄金路径、环境检查、启动命令、兜底策略和故障速查都写清楚了。
Runbook 明确了主演示路径：

这份 Runbook 的意义很大：它把系统从“开发者知道怎么跑”变成“比赛现场可以按脚本稳定展示”。对于最终答辩来说，这比临时靠记忆操作可靠得多。
飞书普通消息体验和安全误判做了修正
本周期还顺手修了一些真实飞书环境里很容易影响演示的小坑：
群聊 @bot 后，会剥离飞书 mention 占位符，避免把 @_user_1 当成用户真实意图。
飞书私聊或明确 @bot 的普通问答，即使 Router 判成 ignore，也会提升为普通 LLM 回复。

这些不是主线功能，但很影响演示观感。它们让机器人更像一个可用助手，而不是只会处理固定任务触发词的接口。

三、过程复盘与沉淀（必填）
1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？
这周的主线是“演示稳定化”。前四个周期已经完成了领域模型、服务编排、文档生成、飞书接入和真实产物链雏形；第五周期要解决的是评委真正看到系统时的体验问题。
最先处理的是 Dashboard。因为之前底层事件和状态已经很完整，所以这次没有重做后端模型，而是把已有的 task、step、approval、artifact、event 重新组织成前端可读的驾驶舱。方法上是直接围绕演示场景拆页面：先看任务、再看审批、再看执行、最后看成果。
第二个重点是飞书卡片审批。原来审批能力已经在 Pilot 内部存在，但演示时用户不会去理解内部 approval object，所以必须把 approval.requested 转成飞书卡片，并且点完按钮后让卡片进入最终态。这里用了 notifier 协议把 Pilot 服务和飞书连接器隔开，避免把飞书实现直接写死在核心服务里。
第三个重点是成果区。Artifact 本来已经具备内容落盘、版本、checksum、share_url 等信息，但如果前端只展示 metadata，用户依然不知道成果是什么。所以这次补了 content preview API，再由 Dashboard 按 artifact type 做卡片化展示。
第四个重点是 Runbook。这个动作看起来像文档工作，但实际是把所有代码能力串成一条可控演示路径：启动前检查、黄金路径、兜底路径、故障速查都写清楚。它让系统从“能跑”变成“能讲”。
2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？
最明显的卡点是飞书审批卡片。飞书卡片不是“发出去就结束”，还涉及按钮回调、长连接事件类型、操作者身份、审批人校验、卡片最终态刷新。尤其是长连接下 SDK 给出的事件类型和 HTTP webhook 事件类型不完全一致，所以需要把 p2.card.action.trigger 归一化为内部统一 payload，再交给已有 webhook processor 处理。
第二个卡点是审批后的续跑。计划审批和 step 审批在业务语义上不同：计划审批通过后可以启动执行；step 审批通过后应该在同一个 task/plan 上继续往后跑，而不是重新创建一个恢复任务。最后通过 ExecutionEngine.resume_plan、TaskOrchestrator.resume_after_step_approval 和 command 层 run_after_approval 参数把这条链路补齐。
第三个卡点是飞书 Drive 链接。真实 API 不一定稳定返回 download_url，如果只依赖这个字段，演示时可能出现“文件上传了但用户拿不到可点链接”的尴尬情况。后来通过 file_token + drive_url_template 做兜底，同时 share_file 给管理员账号，让飞书自己发“与我共享”通知，可靠性高很多。
第四个卡点是“普通飞书消息不该被误伤”。实际演示时经常会发一些普通测试消息，例如“你好，你需要回复我……”。如果 Router 或 Guard 过度谨慎，机器人就像坏了一样。最后通过 mention 清理、addressed ignore promote 和 Guard 降级策略，把这类普通交互兜住了。
3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？
这次最值得复用的是“驾驶舱式展示结构”。以后无论扩展到钉钉、企业微信还是更多 Agent 工作流，都可以沿用这套表达：任务概览、当前活动、待办审批、步骤时间线、成果卡片、事件流。它比直接暴露 JSON 更适合演示，也更适合真实用户理解 Agent 的执行过程。
第二个可复用的是 ApprovalRequestNotifier 抽象。它让审批通知变成一个可替换通道：现在是飞书卡片，后续可以换成钉钉卡片、邮件、Slack、企业微信，而 Pilot 核心只知道“有审批请求需要通知”。
第三个可复用的是 artifact preview API 的边界设计。text/json 允许预览，binary 只返回元数据，这个规则简单、稳定、风险低。未来成果类型增加后，只需要扩展 artifact type 和前端卡片映射，不需要重新设计成果读取机制。
第四个可复用的是 Runbook 驱动的演示方式。先固化黄金路径，再列兜底路径，再列不主推范围，这对比赛项目尤其重要。它能防止现场临时发挥过度，也能把系统真实能力讲得更清楚。
第五周期总体上已经把 Agent-Pilot 推到了“可被观看、可被操作、可被讲清楚”的阶段。剩下的优先级也很明确：PPT 内容还需要更像比赛路演材料，默认生成 6-8 页更精炼；进度查询能力已有基础，但适合放在 UI 主链路稳定后再做后置验证。
