# Agent-Hub 面试答辩手册

## 1. 面试版项目定位

不要把 Agent-Hub 讲成通用 AgentOS。面试时建议收敛成：

> 我做的是一个面向办公场景的 Agent 任务编排与执行轨迹数据化平台。核心不是让模型完全自主行动，而是把 IM 里的模糊需求转成可审批、可追踪、可恢复的结构化任务流，并把执行过程沉淀成后续检索、评估和复盘的数据。

一句话主线：

> 飞书消息进来后，系统先做意图分流和计划生成，再按 DAG 推进步骤；高风险步骤进入审批，所有状态变化写入事件流，最终产物和执行轨迹可在 Dashboard 或飞书端查看。需要特别区分：通用 AgentPipeline 支持同层 DAG 并发，Pilot 业务链路为了审批和恢复一致性，当前同层步骤是顺序推进。

面试中最值得主讲的三件事：

1. 任务执行过程的数据化：Task、Plan、PlanStep、Approval、Artifact、ExecutionEvent 都有明确状态和元数据。
2. 可控任务编排：DAG 分层执行、审批门控、失败阻断、步骤级恢复，不把外部 API 裸露给模型。
3. 检索与评测闭环：pgvector + BM25 + RRF 做混合检索，并用自建 QA 集合评估 Recall@5 / MRR。

不要主讲成：

- 完整通用 Agent 框架。
- 生产级企业中台。
- 全自主多 Agent 协作系统。
- 多 IM 渠道全部落地。
- 成熟安全防护平台。

## 2. 和简历描述的对齐方式

已经投出的简历不能改，所以面试时要把措辞解释成“离线验证 / 本地演示 / 可替换真实链路”，而不是线上生产指标。

| 简历说法 | 面试解释口径 |
| --- | --- |
| 接入飞书的企业办公智能助手 | 飞书 webhook / 长连接 / 审批卡片链路已经做了接口和测试；真实 Drive 分享依赖凭据，演示时保留 fake fallback。 |
| 自动完成任务规划、知识检索、工具调用、审批回调和结果回传 | 主链路由 Pilot 运行时负责；规划是 profile-aware 模板 + LLM 网关增强，审批和状态推进是确定性代码控制。 |
| 积累 100 多条 Agent 执行轨迹 | 来自本地离线演示、测试运行和 SQLite / artifact / Obsidian 记录，不是企业真实用户数据。 |
| 50 组常见办公任务 | 是按办公自动化场景人工构造的模拟任务集合，用来验证通用 AgentPipeline 的 DAG 分层并发调度；不是飞书审批全链路压测。 |
| 80 多条自建办公问答集 | 是自建/改写的评测问题，不含真实企业私有数据；用于比较检索策略，不代表线上问答准确率。 |
| Recall@5 提升约 10% | 指同一批自建问题上，混合召回相对单路向量召回的离线评估结果；样本规模较小，主要证明评估方法和方向。 |

最稳的总回答：

> 这个项目是个人/比赛型项目，没有使用企业真实数据。我把“数据”拆成三类：一类是人工构造的办公任务样本，用来测任务编排；一类是自建知识库 QA，用来测 RAG；一类是系统运行时自动记录的事件轨迹，用来做复盘和定位。简历里的数字都是在这些离线样本和本地演示环境下得到的，我不会把它说成线上生产指标。

## 3. 四个坑点的答辩处理

### 坑点一：边界模糊

推荐回答：

> 我后来把边界收窄了：它不是通用 AgentOS，而是办公任务 Agent 的运行时。主能力是“把一次 IM 任务请求变成可执行、可审批、可观测的数据流”。RAG、安全、飞书、PPT 产物都是围绕这条主线服务的能力，不单独把它们当成一个完整平台去讲。

可以展开：

- 通用 Agent API 是底层能力。
- Pilot 才是面试主线。
- LangGraph 只用于 ReflectionAgent 的检索纠错子流程。
- 钉钉、QQ、OpenClaw 是 connector 预留或占位，不作为已落地能力主讲。

### 坑点二：很多能力是有框架但不够生产级

推荐回答：

> 我没有把它当成生产级系统去吹。这里更像是一个可验证的工程原型：通过模板化计划和确定性状态机保证主链路稳定，再把 LLM 放在路由、计划生成、RAG 重写这些可降级的位置。模型输出不会直接决定写入或分享，所有计划都会经过 Pydantic、DAG 校验、技能白名单和审批策略。

重点承认：

- Planning 很多时候是模板增强，不是完全自由规划。
- Brief 生成不是强内容生产能力，核心价值在任务流和产物链路。
- 测试证明的是控制流、状态流、事件流可跑，不证明真实 Agent 质量已经可靠。

然后转优点：

> 我认为这是一个有意的工程取舍：面试项目里比“看似智能但不可控”更重要的是，把状态、审批、恢复和评测口径做清楚。

### 坑点三：安全模块容易被追问

推荐回答：

> 安全部分我会低调讲成“风险意识和入口防护”，不是成熟安全系统。当前实现是规则层 + 可选 LLM 语义检测，规则层有小规模攻击样本脚本能看拦截率和误杀率，但样本量有限，也没有红队级绕过评估。生产上还需要持续样本收集、误杀回放、成本延迟评估和策略灰度。

可讲价值：

- 规则命中直接拦截，减少 LLM 成本。
- LLM 异常时降级，避免阻塞正常业务。
- 写入/分享等高风险动作不靠 prompt guard，而靠审批门控和技能权限。

不要说：

- 已经全面防住 prompt injection。
- 达到生产安全标准。
- 有完整攻防体系。

### 坑点四：Agent 味不够，Workflow 味重

推荐回答：

> 这个判断是对的。我的项目不是“完全自主多 Agent 协商”，而是 workflow-controlled agent。也就是说，我用 LLM 做意图识别、计划补全、检索重写和内容生成，但执行层是确定性的 DAG、状态机和审批策略。原因是办公场景里可控性比自主性更重要，尤其是涉及文档写入、分享、外部平台 API 时。

如果追问 Agent 在哪里：

- LLM Router：把自然语言转为结构化 RoutingDecision / PlanProfile。
- RAG Reflection：retrieve -> evaluate -> rewrite -> generate 的纠错流程。
- Tool/Skill 调用：技能由元数据声明副作用和审批要求。
- 失败恢复：现阶段是步骤级 retry/resume，不是自动重规划。

边界回答：

> 还没有做多 Agent 协商、冲突解决和基于中间结果的动态重规划。这些我会放到下一步，比如把失败原因分类后触发局部 replan，而不是重新跑完整任务。

## 4. 数据来源与评测口径

### 轨迹数据怎么来的

回答：

> 轨迹来自系统运行时自动记录的事件，不是手写日志。每次任务会产生 task.created、plan.generated、step.status_changed、approval.requested、artifact.created 等事件。事件里包含 task_id、plan_id、step_id、trace_id、status、risk_level、artifact_id 等字段，所以可以从一条 trace 复盘任务是怎么走到成功、失败或 blocked 的。

补充：

- SQLite 和 artifact 文件用于本地持久化演示。
- Obsidian markdown 记录来自旧 memory 模块。
- 100 多条是本地反复跑 demo、测试和离线脚本产生的样本，不是线上用户日志。

### “10 分钟缩短到 2 分钟”怎么测

回答：

> 这是一个离线人工排障对比，不是线上 SLA。我选了几类常见问题，比如步骤失败、审批卡住、上游产物缺失、重复事件。没有结构化事件前，需要翻应用日志和中间状态，大概 10 分钟才能定位；有 trace / event timeline 后，通常 2 分钟内能定位到具体 step、状态和错误信息。这个数字主要表达可观测性带来的定位效率提升。

最好主动补一句：

> 如果严格做实验，应该固定问题集、多人重复测量、记录均值和方差；我目前只是个人项目中的对比记录，所以面试里会如实说明。

### “45% 耗时降低”怎么测

回答：

> 我构造了有依赖关系的办公任务 DAG，把可并行步骤放在同一拓扑层，用 asyncio.gather 并发执行；对比方式是同一批模拟任务在串行执行和 DAG 分层执行下的端到端耗时。这个指标主要反映 I/O 型步骤并行化收益，比如检索、上下文读取、文档生成前置处理可以并行。

更精确的面试口径：

> 仓库里要分两层看：`core/pipeline.py` 的通用 `AgentPipeline` 确实用了 Kahn 拓扑分层和 `asyncio.gather`，我用 `scripts/benchmark_dag.py` 跑 50 组模拟办公 DAG，生成 `reports/benchmark_dag.md/json` 来复现这个耗时下降。Pilot 的 `ExecutionEngine` 也会按 DAG 分层，但同层 step 是顺序推进，因为审批、Artifact 写入和 blocked/resume 都是业务状态，贸然并发会让一次任务出现多个待审批点，恢复语义变复杂。所以“45%”说的是通用编排内核调度收益，不是飞书审批全链路端到端 SLA。

当前本地报告示例：50 组模拟办公 DAG，串行平均约 356 ms，DAG 并发平均约 138 ms，缩短约 61%。面试时不要把 61% 当固定收益，可以说“在模拟 I/O 型任务里稳定超过 45%，具体取决于 DAG 并行宽度和外部 API 限流”。

边界：

> 这个不是 LLM 质量提升，也不是所有任务都能提升 45%。如果 DAG 依赖链很长，收益会下降；如果外部 API 有严格限流，还要加并发控制。

### “Recall@5 提升约 10%”怎么测

回答：

> 我先构造文档和 QA 样本，每条问题标注 relevant_doc_ids，然后分别跑单路向量检索和混合检索。混合检索先取 dense 和 BM25 两路结果，再用 RRF 融合排序，最后用 Recall@5、Precision@5、MRR 做离线评估。提升 10% 指的是这批自建样本上的相对提升。

边界：

> 样本量不大，而且很多问题是围绕项目知识和办公知识构造的，所以它证明的是评测闭环和方法有效，不代表泛化到所有企业知识库都稳定提升。

## 5. 面向腾讯游戏数据方向的讲法

腾讯游戏数据部门可能更关心数据链路和业务可观测性，可以这样转译：

> 虽然项目场景是办公 Agent，但我真正做的是把一次自然语言任务转成结构化执行数据，再基于这批数据做状态追踪、失败定位、检索评估和策略迭代。这个思路和游戏数据里的埋点、链路追踪、指标归因、异常定位是相通的。

可以重点讲：

- 事件模型：把复杂过程拆成标准事件，而不是只存最终结果。
- trace_id：串联用户请求、计划、步骤、审批、产物。
- 指标口径：耗时、成功率、blocked 率、失败类型、可追溯率。
- 检索评估：构造标注集，用 Recall@K / MRR 衡量策略变化。
- LLM 使用边界：LLM 适合做语义理解和召回增强，不适合直接绕过权限执行动作。

避免讲太多：

- “自动生成 PPT”。
- “多 Agent 自主协作”。
- “企业级安全中台”。

## 6. 面试开场讲稿

推荐 60 秒版本：

> Agent-Hub 是我做的一个办公任务 Agent 运行时。它解决的问题不是单纯聊天，而是用户在飞书里说一句比较模糊的需求后，系统怎么把它变成可执行、可审批、可追踪的任务流程。
>
> 我的主线是三层：第一层是入口和规划，把 IM 消息转成 Task、Plan 和 PlanStep；第二层是执行和治理，用 DAG 拓扑分层调度步骤，并对写入、分享这类高风险动作做审批门控和失败恢复；第三层是数据闭环，所有状态变化都写 ExecutionEvent，用 trace_id 串起来，Dashboard 和飞书端看到的是同一条事实流，同时这些轨迹也能用于后续定位和复盘。
>
> RAG 这块我做了 pgvector 向量检索、BM25 稀疏检索和 RRF 融合，并用自建 QA 集合做 Recall@5 / MRR 评估。需要说明的是，这个项目是个人/比赛环境，没有企业真实数据，所以简历里的轨迹和评测指标都是离线模拟样本上的结果，我会把它定位成工程原型，而不是线上生产系统。

## 7. 最容易被问的问题

### Q1：你这个项目到底最核心的技术难点是什么？

> 不是调用 LLM，而是把不稳定的 LLM 输出放进一个可控的工程闭环里。模型可以生成计划，但计划必须被结构化校验；模型可以建议调用工具，但工具有 SkillSpec、side_effect、approval 和 scope；任务可以失败，但状态机和事件流要能定位失败点并支持恢复。

### Q2：既然是模板规划，为什么还叫 Agent？

> 我会叫它 workflow-controlled agent。办公场景里，让模型完全自由规划并不安全，所以我把自主性限制在语义理解、计划补全、RAG 重写和内容生成上；执行层保持确定性。这个取舍牺牲了一些“智能感”，换来可解释、可审批和可恢复。

### Q3：数据都是假的，指标可信吗？

> 我不会说它代表真实业务效果。它可信的地方是评测流程可复现：样本怎么构造、怎么标注、指标怎么算、对比对象是什么都是固定的。它不可信或者说还没验证的是线上泛化能力，这需要真实业务日志和更大规模标注集。

### Q4：如果接到真实业务，你下一步怎么做？

> 我会先做数据层补强：定义真实事件 schema、接入脱敏日志、建立评测集版本管理，然后把现在的模拟指标替换成真实指标。功能上优先做失败原因分类、局部重规划、检索样本沉淀和权限策略审计，而不是继续堆更多 connector。

### Q5：安全怎么评估？

> 当前只有小规模规则样本和单测，能证明入口拦截流程存在，但不能证明安全效果成熟。真实落地要补攻击样本集、正常样本集、误杀率、漏放率、延迟、成本和绕过案例回放。更重要的是，高风险外部动作不能依赖 prompt guard，而要依赖权限、审批和可审计执行策略。
